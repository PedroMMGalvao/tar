Este trabalho implementa um canal de comunicação encoberto, in-\/band, sobre Ethernet L2, usando {\bfseries{AF\+\_\+\+XDP zero-\/copy}} e {\bfseries{e\+BPF/\+\+XDP em modo native DRV}}. Ao nível da segurança, é aplicada a cifra {\bfseries{AES-\/256-\/\+CTR com HMAC-\/\+SHA256}}. Desenvolvido no âmbito da unidade curricular Advanced Network Topics (MERSI, FCUP).

\hypertarget{md_README_autotoc_md4}{}\doxysection{\texorpdfstring{Visão geral}{Visão geral}}\label{md_README_autotoc_md4}
O canal transmite tramas Ethernet em broadcast de {\bfseries{270 bytes}} com Ether\+Type {\ttfamily 0x88\+B5} (IEEE private use, sem significado IANA). Assim, cada trama transporta um payload de 256 bytes ({\ttfamily \doxylink{structstealth__pkt__t}{stealth\+\_\+pkt\+\_\+t}}) com magic {\ttfamily "{}STLH"{}}, número de sequência, IV aleatório, texto cifrado AES-\/256-\/\+CTR e tag HMAC-\/\+SHA256.

O programa XDP ({\ttfamily \doxylink{xdp__kern_8c}{xdp\+\_\+kern.\+c}}), carregado em modo native DRV no driver {\ttfamily mlx5}, permite:\+
\begin{DoxyItemize}
\item filtragem das tramas por Ether\+Type;
\item validação do magic;
\item aplicação de anti-\/replay via {\ttfamily replay\+\_\+map} (LRU hash);
\item redirecionamento das tramas para o socket AF\+\_\+\+XDP via {\ttfamily bpf\+\_\+redirect\+\_\+map}.
\end{DoxyItemize}

Nos hosts sem o programa XDP correspondente, as tramas são descartadas silenciosamente pelo kernel por terem um Ether\+Type desconhecido.

A API {\ttfamily EVP\+\_\+\+MAC} do Open\+SSL 3.\+x é usada em modo incremental:\+ {\ttfamily EVP\+\_\+\+MAC\+\_\+update(ctx, iv, 16)} seguido de {\ttfamily EVP\+\_\+\+MAC\+\_\+update(ctx, ct, len)}. Assim, evita-\/se alocar um buffer apenas para concatenar IV e CT.
